# Документооборот агентства: УПД, счета, договоры (нормы РФ)

Первоисточник правил для ИИ. Действует для обоих профилей: ИП Малышев М.А. и
ИП Абилвапов Э.М. Оба — **УСН 6%, без НДС** (п. 2 ст. 346.11 НК РФ).

```
verified: 2026-07-22      # дата последней сверки норм с первоисточниками
review_after: 90 дней     # старше → сначала актуализация (§7), потом документы
```

При работе со счетами/закрывающими/договорами ИИ обязан следовать этому файлу.
Если `verified` старше `review_after` ИЛИ есть любое сомнение в норме —
сначала процедура актуализации (§7), только потом создание документов.

## 1. Закрывающие документы

**Основной закрывающий — УПД (функция ДОП).** Акты для обмена через ЭДО не выставлять.

Нормативка:
- С **01.01.2026** через операторов ЭДО запрещён обмен по форматам ТОРГ-12
  (приказ ФНС № ММВ-7-10/551@) и акта выполненных работ (№ ММВ-7-10/552@) —
  приказ ФНС от 20.01.2025 **№ ЕД-7-26/28@**, информация ФНС от 07.10.2025.
- Единственный формализованный формат — **УПД 5.03** (приказ ФНС от 19.12.2023
  № ЕД-7-26/970@, обязателен с 01.04.2025).
- Акт как вид первички НЕ отменён (ст. 9 закона 402-ФЗ): бумажный и
  неформализованный электронный акт законны, но в контуре ЭДО не используем.

Правила для ИИ:
- Мы без НДС ⇒ УПД только **функция ДОП / статус 2** (первичка-аналог акта).
  **СЧФДОП / статус 1 — запрещено**: включает счёт-фактуру, неплательщик НДС её
  не выставляет (риск: у клиента ошибочный вычет, у нас — вопросы ИФНС).
- Инструмент: `bill.py upd <client> --amount N [--bill <documentId счёта>]`
  (`~/eldar_progress/scripts/tochka/bill.py`, Точка API Create Closing Document,
  `Content.Upd`, `Positions[].function = "dop"`).
- `bill.py act` — только если клиент просит бумажный/неформализованный акт.

## 2. Сроки и периодичность (ст. 9 402-ФЗ, п. 3)

- Первичный документ составляется **при совершении факта хозяйственной жизни или
  непосредственно после**. Закрывающие задним числом (документ в июле за работы
  апреля) — не делать; если просрочили — дата документа = текущая, период услуг
  указывается в названии позиции.
- **Абонентское обслуживание** (длящаяся услуга) — один УПД в месяц, последним
  числом месяца, за все услуги периода (допустимо по разъяснениям Минфина для
  непрерывных услуг: письма от 12.01.2016 № 03-07-09/140, от 02.05.2012 № 03-07-09/44).
- **Разовые работы/этапы** (разработка, доработка) — отдельный УПД на дату
  сдачи каждого этапа. Объединять этапы с разными датами приёмки в один
  документ нельзя.
- Несколько услуг, принятых одной датой, одному клиенту — один УПД с несколькими
  позициями (строками), не пачка документов.

## 3. Счета

- Счёт на оплату — не первичный документ, формы/срока по закону нет, но:
  номер счёта — в назначении платежа клиента (Точка матчит оплату по ИНН + сумме +
  номеру счёта), поэтому счёт всегда ДО оплаты, закрывающий — ссылкой на счёт (`--bill`).
- В счёте и УПД: основание = реквизиты договора (`basedOn` из clients.json,
  формат «Договор № N от DD.MM.YYYY»), НДС = «Без НДС» (не «НДС 0%» — это разные вещи:
  0% — экспортная ставка со счётом-фактурой, «без НДС» — УСН).
- Отправка счёта клиенту — только после «ок» владельца (правило скилла tochka).

## 4. Аннулирование и исправления (ЭДО)

- Подписанный обеими сторонами электронный документ удалить нельзя — только
  **аннулирование по соглашению сторон** (двустороннее: предложение + подпись
  контрагента). Висящие «В процессе аннуляции» — контролировать и напоминать.
- Ошибка в подписанном УПД → исправленный УПД (УКД для корректировки
  стоимости), не «перевыставить новый как ни в чём не бывало».
- Замена уже подписанных корректных закрывающих «одним общим» — не делать
  (см. §2: разные факты хозжизни + документ задним числом).

## 5. Договоры

- Работы для юрлиц/ИП — только при наличии договора; реквизиты договора заводятся
  в `clients.json` (`basedOn`) ДО выставления первого счёта.
- Письменная форма обязательна для сделок юрлиц/ИП между собой (ст. 161 ГК РФ);
  обмен подписанными сканами/ЭДО допустим (п. 2 ст. 434 ГК РФ), оферта-акцепт
  (счёт + оплата) — запасной вариант, но для абонентки всегда полноценный договор.
- В договоре фиксировать: предмет, цену, порядок сдачи-приёмки (какими
  документами закрываемся — прописать УПД), сроки оплаты.
- Реестр действующих договоров: `docs/ai-coo/contracts.yml`.

## 6. Чек-лист ИИ перед созданием любого документа

1. Есть договор? (`clients.json.basedOn`, `contracts.yml`) — нет → сначала договор.
2. Тип закрывающего = УПД ДОП. Акт — только по явной просьбе, вне ЭДО.
3. Дата документа = дата факта (сдачи работ / последний день месяца для абонентки).
4. НДС = without_nds. Никаких счетов-фактур и СЧФДОП.
5. Закрывающий привязан к счёту (`--bill`).
6. Отправка/подпись — только владелец (T2-тир).

## 7. Актуализация (когда `verified` просрочен или законы могли измениться)

Прогнать поиском (web_search/firecrawl) и сверить с текущим текстом файла:

1. **Форматы ЭДО**: «форматы электронных документов ФНС УПД действующий приказ
   <текущий год>» — не сменился ли формат УПД 5.03 (№ ЕД-7-26/970@) и не появились
   ли новые обязательные форматы. Первоисточники: nalog.gov.ru → «Перечень
   форматов, утверждённых ФНС», changelog Точка API (developers.tochka.com).
2. **УСН и НДС**: «УСН НДС изменения <год> порог освобождение» — статус
   «без НДС» обоих ИП (порог доходов для НДС на УСН меняется! в 2026 — см.
   `tax-profile.md`). Если ИП становится плательщиком НДС — §1 переписать:
   потребуются счета-фактуры/СЧФДОП, правило «статус 1 запрещён» снимается.
3. **Первичка**: изменения в ст. 9 402-ФЗ и ФСБУ 27/2021 — «402-ФЗ первичные
   документы изменения <год>».
4. **Точка API**: https://developers.tochka.com/docs/tochka-api/changelog —
   не изменились ли методы Create Invoice / Create Closing Document (`bill.py`
   завязан на них).

По результатам: обновить текст затронутых секций, поставить свежую дату в
`verified`, изменения нормативки — флагом владельцу в брифинг. Если изменение
требует правок `bill.py`/`monthly.py` — сделать в тот же заход.

Плановые точки контроля:
- **Еженедельно** (в рамках weekly-planning): Firecrawl-монитор
  «doc-flow: форматы ФНС + Точка API», id `019f8985-4001-727a-a090-c2a732fb14ff`,
  сам снимает обе страницы (перечень форматов ФНС + changelog Точка API)
  каждое воскресенье. На планировании: `firecrawl_monitor_checks` по этому id →
  если есть `changed` с осмысленным вердиктом judge — прогнать §7 полностью
  и обновить этот файл.
- **Первый понедельник квартала** (январь/апрель/июль/октябрь): полный прогон
  §7 принудительно, даже если монитор молчал и `verified` не просрочен —
  монитор покрывает только две страницы, изменения НК/402-ФЗ он не видит.
